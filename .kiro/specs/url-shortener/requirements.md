# 需求文档

## 简介

短链接服务是一个基于 Django REST Framework 构建的 API 后端服务，用于将长 URL 转换为短链接，并提供访问统计功能。系统采用 MySQL 存储数据，Redis 缓存热点链接，支持 JWT 认证和接口限流。

## 术语表

- **短链接服务**: 系统主体，负责处理所有短链接相关的业务逻辑
- **短码**: 由 Base62 字符（a-z, A-Z, 0-9）组成的唯一标识符
- **原始链接**: 用户提交的需要缩短的长 URL
- **已认证用户**: 已注册并通过 JWT 认证的系统用户
- **匿名用户**: 未登录的访客
- **限流器**: 接口限流组件，用于控制 API 请求频率
- **缓存层**: Redis 缓存，用于存储热点短链接数据

## 需求列表

### 需求 1：用户认证

**用户故事：** 作为用户，我希望能够注册和登录系统，以便管理我的短链接并使用高级功能。

#### 验收标准

1. 当用户提交有效的注册信息（用户名、邮箱、密码）时，短链接服务应创建新用户账户并返回成功响应
2. 当用户提交无效的注册信息时，短链接服务应返回相应的错误信息说明验证失败原因
3. 当已注册用户提交正确的登录凭证时，短链接服务应返回 JWT 访问令牌和刷新令牌
4. 当用户提交错误的登录凭证时，短链接服务应返回认证错误
5. 当用户提供有效的刷新令牌时，短链接服务应返回新的访问令牌
6. 当用户提供过期或无效的刷新令牌时，短链接服务应返回认证错误

### 需求 2：短链接创建

**用户故事：** 作为用户，我希望能够将长链接转换为短链接，以便更方便地分享。

#### 验收标准

1. 当已认证用户提交有效的原始链接时，短链接服务应生成唯一短码并返回完整的短链接
2. 当已认证用户提交无效的链接格式时，短链接服务应返回验证错误
3. 当已认证用户提交的链接在其账户中已存在时，短链接服务应返回已有的短链接
4. 当已认证用户提供自定义短码时，如果该短码可用且有效，短链接服务应使用该短码
5. 当已认证用户提供的自定义短码已被占用时，短链接服务应返回冲突错误
6. 当使用自定义短码功能时，短链接服务应验证短码仅包含 Base62 字符且长度在 4-10 个字符之间
7. 当短链接创建成功时，短链接服务应将映射关系存储到 MySQL 并缓存到 Redis

### 需求 3：短链接重定向

**用户故事：** 作为访客，我希望访问短链接时能够跳转到原始链接，以便快速到达目标页面。

#### 验收标准

1. 当访客访问有效的短码时，短链接服务应以 HTTP 302 状态码重定向到原始链接
2. 当访客访问不存在的短码时，短链接服务应返回 HTTP 404 错误
3. 当访客访问有效的短码时，短链接服务应优先从缓存层查询，缓存未命中时再查询数据库
4. 当短码被访问时，短链接服务应异步增加访问计数
5. 当短码被访问时，短链接服务应记录访问时间戳、IP 地址和 User-Agent

### 需求 4：短链接管理

**用户故事：** 作为已认证用户，我希望能够管理我的短链接，以便查看、更新和删除它们。

#### 验收标准

1. 当已认证用户请求链接列表时，短链接服务应返回分页的短链接列表及统计信息
2. 当已认证用户请求特定短链接详情时，短链接服务应返回链接详情及访问统计
3. 当已认证用户更新短链接的原始链接时，短链接服务应更新映射关系并使缓存失效
4. 当已认证用户删除短链接时，短链接服务应移除映射关系并使缓存失效
5. 当已认证用户尝试访问其他用户的短链接时，短链接服务应返回禁止访问错误

### 需求 5：访问统计

**用户故事：** 作为已认证用户，我希望查看短链接的访问统计，以便了解链接的使用情况。

#### 验收标准

1. 当已认证用户请求短链接统计时，短链接服务应返回总点击量、独立访客数和最近访问记录
2. 当已认证用户请求时间维度统计时，短链接服务应返回指定日期范围内按天分组的点击量
3. 当请求统计数据时，短链接服务应高效地从访问日志中聚合数据

### 需求 6：接口限流

**用户故事：** 作为系统管理员，我希望限制 API 请求频率，以便系统保持稳定并对所有用户公平。

#### 验收标准

1. 当用户超过请求频率限制时，限流器应返回 HTTP 429 状态码并包含 retry-after 响应头
2. 限流器应对已认证用户（100 次/分钟）和匿名用户（20 次/分钟）应用不同的限制
3. 限流器应使用 Redis 配合滑动窗口算法来追踪请求计数
4. 当接近限流阈值时，短链接服务应在响应头中包含剩余配额信息

### 需求 7：API 文档

**用户故事：** 作为开发者，我希望能够访问 API 文档，以便了解如何与服务集成。

#### 验收标准

1. 短链接服务应在 /api/docs/ 端点提供 Swagger/OpenAPI 文档
2. 短链接服务应为所有端点提供请求/响应示例
3. 短链接服务应为每个端点标注认证要求

### 需求 8：链接过期管理

**用户故事：** 作为已认证用户，我希望为短链接设置过期时间，以便控制分享链接的有效期。

#### 验收标准

1. 当已认证用户创建带过期时间的短链接时，短链接服务应存储过期时间戳
2. 当访客访问已过期的短码时，短链接服务应返回 HTTP 410 Gone 状态码并显示适当提示
3. 短链接服务应运行定时任务清理缓存中的过期链接
4. 当已认证用户更新短链接时，短链接服务应允许修改或移除过期时间

### 需求 9：批量操作

**用户故事：** 作为已认证用户，我希望能够批量创建和管理短链接，以便提高工作效率。

#### 验收标准

1. 当已认证用户提交一批链接（最多 50 个）时，短链接服务应为所有有效链接创建短链接并返回每个链接的处理结果
2. 当批量创建包含无效链接时，短链接服务应处理有效链接并为无效链接返回单独的错误信息
3. 当已认证用户请求批量删除时，短链接服务应删除用户拥有的所有指定链接
4. 当批量操作超过 10 个项目时，短链接服务应异步处理请求

### 需求 10：链接分组与标签

**用户故事：** 作为已认证用户，我希望用分组和标签来组织短链接，以便更有效地管理它们。

#### 验收标准

1. 当已认证用户创建分组时，短链接服务应存储分组名称和可选描述
2. 当已认证用户将短链接分配到分组时，短链接服务应更新链接的分组关联
3. 当已认证用户为短链接添加标签时，短链接服务应存储标签关联
4. 当已认证用户按分组或标签筛选链接时，短链接服务应返回匹配的分页链接列表
5. 当已认证用户删除分组时，短链接服务应移除分组但保留链接（将分组设为空）

### 需求 11：安全防护

**用户故事：** 作为系统管理员，我希望保护服务免受恶意使用，以便平台保持安全可信。

#### 验收标准

1. 当用户提交链接时，短链接服务应将链接与恶意域名黑名单进行比对验证
2. 当检测到潜在恶意链接时，短链接服务应拒绝请求并记录该尝试
3. 短链接服务应实施 CORS 策略以限制跨域请求
4. 当短链接收到异常高流量时，短链接服务应触发告警并可选择性禁用该链接
5. 短链接服务应对所有用户输入进行清理以防止 XSS 和 SQL 注入攻击

### 需求 12：数据导出

**用户故事：** 作为已认证用户，我希望导出我的链接数据，以便备份或在外部进行分析。

#### 验收标准

1. 当已认证用户请求数据导出时，短链接服务应生成包含所有链接和统计信息的 CSV 文件
2. 当导出大量数据时，短链接服务应异步处理导出请求并在完成后通知用户
3. 短链接服务应在导出文件中包含链接详情、创建日期、过期日期、点击量和标签
